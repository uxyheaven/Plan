# AI 分析系统设计方案

## 1) 架构设计（模块精简介绍）

```text
[卡片入口/UI层]
   ├─ 卡片聚合服务（基础指标、告警、入口动作）
   └─ 对话工作台（查看报告、追问、直接提问）
            │
            ▼
[分析编排层]
   ├─ 场景路由器（Scene Router）
   ├─ 任务规划器（Planner）
   ├─ 框架选择器（Framework Selector）
   ├─ 数据缺口检测器（Data Gap Detector）
   ├─ 工具调度器（Tool Orchestrator）
   └─ 证据与引用管理器（Evidence Manager）
            │
            ▼
[执行与计算层]
   ├─ 指标查询执行器（Metric Engine）
   ├─ API 连接器（Connector Hub）
   ├─ SQL 执行器（SQL Runner）
   ├─ 规则/统计计算引擎（Deterministic Compute）
   └─ 洞察生成器（Insight Synthesizer）
            │
            ▼
[模型与知识层]
   ├─ LLM 网关（Prompt模板、调用策略、审计）
   ├─ 场景库（Scene Registry）
   ├─ 分析框架库（Framework Registry）
   ├─ Skill 库（能力插件）
   └─ 元数据目录（指标定义、口径、权限、血缘）
            │
            ▼
[治理与运维层]
   ├─ 观测（日志、Trace、Token、耗时）
   ├─ 准确性守护（禁止LLM计算、结果校验）
   ├─ 权限与脱敏（RBAC/ABAC）
   └─ 配置中心（场景、框架、提示词版本化）
```

### 关键模块说明
- **场景路由器**：根据卡片上下文、用户问题、业务域标签识别“分析场景”。支持配置新增场景（如经营诊断、流量波动、销售漏斗、成本异常）。
- **任务规划器**：LLM只负责“规划”，输出步骤树（假设→验证→结论），不做数值计算。
- **框架选择器**：从可用分析框架中选最合适的（可多选），如 AARRR、漏斗分析、同比环比、归因树、5Why。
- **数据缺口检测器**：检查每一步所需数据是否齐全；若缺失，触发工具调度（API/指标/SQL）。
- **确定性计算引擎**：统一做聚合、同比环比、显著性检验、贡献度拆解，保证可复现与准确性。
- **证据与引用管理器**：每个结论绑定“数据来源、查询语句/指标ID、时间窗、版本、提取时间”。
- **洞察生成器**：把计算结果映射为“发现-原因-影响-建议”，并标注置信度与证据链。

## 2) 数据流

1. **入口阶段（卡片）**：卡片展示核心指标与异常信号，用户点击“分析”或直接发问。
2. **理解阶段**：场景路由器识别场景；结合用户意图加载对应场景提示词与候选框架。
3. **规划阶段**：LLM生成分析计划（步骤、假设、所需数据、预期输出）。
4. **执行阶段**：
   - 数据缺口检测器检查可用数据；
   - Tool Orchestrator 调用 API/指标系统/SQL；
   - 计算引擎执行全部数值计算与统计检验。
5. **合成阶段**：洞察生成器产出分解结果（分层、分群、归因）+ 洞察（why/so what/next step）。
6. **引用阶段**：Evidence Manager 给每条结论挂出处（source_id、query_id、metric_id、time_range）。
7. **交互阶段**：用户可追问；系统复用上下文与中间结果增量分析，避免重复取数。

## 3) 每次 LLM 调用的精简介绍

> 原则：**LLM参与理解、规划、解释；不参与计算。**

1. **调用A：意图+场景识别**
   - 输入：用户问题、卡片上下文、历史会话摘要。
   - 输出：场景ID、任务类型（诊断/解释/预测性讨论）、约束条件。

2. **调用B：分析计划生成**
   - 输入：场景模板、候选分析框架、可用数据目录。
   - 输出：步骤化计划（Step DAG）、每步所需数据、优先级。

3. **调用C：框架选择与裁剪**
   - 输入：场景目标、计划草案、框架库描述。
   - 输出：主框架+辅框架，以及适用原因（可审计）。

4. **调用D：结果解释与洞察文本化**
   - 输入：计算引擎产出的结构化结果、证据链。
   - 输出：自然语言结论、洞察、建议动作（附证据引用锚点）。

5. **调用E：追问应答（增量）**
   - 输入：用户追问、已有中间结果、引用记录。
   - 输出：直接回答或新增子任务计划（若数据不足则触发取数）。

## 4) 扩展性设计

### 4.1 场景可扩展
- 场景定义采用 `Scene DSL/JSON`：
  - `scene_id`、适用问题模式、默认提示词、候选框架、必需指标、数据权限策略。
- 新增场景 = 配置+少量校验脚本，不改主流程代码。

### 4.2 分析框架可扩展
- 框架定义采用 `Framework Spec`：
  - 输入要求、步骤模板、可复用计算算子、适用边界。
- 框架可组合（主框架+子框架），并支持按场景白名单。

### 4.3 数据源可扩展
- 统一 Connector 接口：`discover() / fetch() / schema() / lineage()`。
- 支持新增：REST API、指标平台、OLAP、直连SQL、文件仓。
- 所有源都归一到统一中间数据模型（UDM），便于计算引擎复用。

### 4.4 Prompt/Skill 可扩展
- Prompt 模板版本化管理（灰度发布、A/B）。
- Skill 插件化：如“漏斗诊断skill”“库存归因skill”“异常检测skill”。
- 通过技能注册中心声明输入输出契约，支持热插拔。

### 4.5 准确性与治理可扩展
- 规则校验插件：口径冲突检查、异常值检查、数据时效检查。
- 审计链路：每次分析生成可回放包（prompt、计划、查询、结果、引用）。

## 5) Skill 与场景、分析框架的关系

- **场景（Scene）**：回答“在什么业务语境下分析”。
- **分析框架（Framework）**：回答“按什么方法论分析”。
- **Skill**：回答“具体怎么执行某类专门任务（工具+流程+规则）”。

关系可理解为：

`Scene（业务上下文） -> 选择 Framework（方法论） -> 调用 Skill（可执行能力单元）`

示例：
- 场景=“电商转化下滑”
- 框架=“漏斗分析 + 归因树”
- Skills=
  1) `funnel_breakdown_skill`（分渠道/端/版本拆解）
  2) `campaign_api_fetch_skill`（拉取投放数据）
  3) `sql_cohort_skill`（跑队列留存SQL）
  4) `insight_rank_skill`（按影响度排序洞察）

即：**场景决定方向，框架决定结构，skill决定执行深度与效率**。

## 最核心的 3 个问题与解决思路

### 核心问题1：如何在多场景下做到“可控地自主分析”？
**解决思路**：
- 用“场景路由 + 计划生成 + 计划约束校验”三段式。
- 计划必须结构化（Step DAG），并通过规则引擎校验（是否可取数、是否越权、是否缺关键步骤）。
- 允许自主规划，但不允许自由执行未授权工具。

### 核心问题2：如何保证准确性（尤其数值）？
**解决思路**：
- 严格执行“LLM不计算”，所有计算进入确定性计算引擎。
- 指标口径统一由元数据目录管理，查询前先做口径绑定。
- 输出前执行自动对账（抽样复算、边界检查、同比环比一致性检查）。

### 核心问题3：如何让结论可信、可追问、可追责？
**解决思路**：
- 每条结论强制绑定出处（指标ID/SQL/API、时间窗、版本、提取时间）。
- 会话内保存中间结果与证据链，追问时优先复用，减少漂移。
- 全链路审计与可回放，支持复核“当时为何得到该结论”。

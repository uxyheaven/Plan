# 分析洞察系统设计规范

本规范描述“分析洞察系统”架构、数据流、LLM 调用策略、可扩展性设计，以及技能（skill）与场景/分析框架的关系。

---

## 1. 架构设计（模块划分）

### 1.1 卡片入口层（Card UI）

- **作用**：作为分析系统的入口，以“卡片”形式承载一个场景的基本数据与交互入口。
- **职责**：展示关键数据摘要、配置场景/分析框架、触发分析、展示结果与对话历史（问答式查看/追问）。
- **特点**：轻量、嵌入式、可复用（可作为微前端嵌入到大盘或 BI 平台中）。

### 1.2 场景与分析管理层（Scene & Framework Manager）

- **作用**：管理“场景”和“分析框架”的定义与生命周期。
- **职责**：
  - 支持新增/编辑/选择场景（包含默认提示词、自定义提示词、可用数据源）。
  - 支持新增/编辑/选择分析框架（例如：漏斗分析、趋势分析、异常检测、关联分析、因果推理等）。
  - 路由：根据用户选择与系统规划，将任务分配给分析引擎。

### 1.3 数据编排与接入层（Data Orchestration）

- **作用**：负责从不同系统获取所需数据，并对数据出处进行标注。
- **职责**：
  - 统一抽象各种数据源：API、指标存储、直连 SQL、文件（CSV/Parquet）等。
  - 提供“数据来源描述/元信息”（例如：数据表名、接口地址、指标 ID、更新时间、取数条件）。
  - 提供缓存与权限控制（避免重复调用、控制访问）。

### 1.4 分析引擎（Analysis Engine）

- **作用**：根据场景与选定的分析框架执行数据处理、指标计算、洞察生成。
- **职责**：
  - 调度分析步骤（规划执行序列）：初始化 -> 数据获取 -> 数据处理 -> 输出洞察。
  - 执行非 LLM 的计算逻辑（统计、聚合、指标计算、模型推断等）。
  - 与 LLM 交互获取洞察文本和步骤建议（但不让 LLM 做数值计算）。

### 1.5 LLM 调用与提示层（LLM Orchestration）

- **作用**：负责与 LLM 交互，负责生成洞察的自然语言输出、规划分析步骤、生成解释、回答用户追问。
- **职责**：
  - 构建调用 prompt：包括场景、分析框架、数据来源/指标说明、计算结果摘要、已执行步骤。
  - 控制调用频率与成本（如分段调用、缓存常见结果）。
  - 处理 LLM 输出，提取可执行指令（如需要补充数据、调整分析维度等）。

### 1.6 交互与追问层（Conversation Layer）

- **作用**：支撑用户与分析系统的对话式交互，支持“直接查看结果并追问”或“直接问（从零开始）”。
- **职责**：
  - 记录会话上下文，维护问题/回答历史。
  - 将用户追问映射到当前场景/分析任务，追加到 LLM 调用中的上下文。
  - 处理“直接问”的场景：在没有场景限制时，自动匹配或推荐场景与分析框架。

## 2. 数据流（核心路径）

1. 卡片启动：用户在卡片上选择/新增场景 + 选定分析框架。
2. 场景管理层生成场景配置：包括默认提示词、可用数据源、分析目标。
3. 分析引擎开始规划步骤：确定需要哪些数据指标，以及哪些框架/方法。
4. 数据编排层获取数据：依次调用 API/SQL/指标平台，返回数据并附带“出处元信息”。
5. 分析引擎执行计算（本地/服务端）：生成指标、聚合、趋势/过滤等结果。
6. LLM 负责“洞察文本”与“分析步骤说明”：将计算结果、数据来源、场景/框架信息传给 LLM，返回自然语言的洞察与推荐。
7. 卡片展示结果（表格/图表 + 洞察文本 + 数据出处）并可进入“对话追问”模式。
8. 追问时：前端把会话上下文与最新计算结果传给 LLM 层，LLM 返回补充说明或新增分析建议；必要时分析引擎获取新增数据再执行一次。

## 3. 每次 LLM 调用的精简介绍

### 3.1 目标：规划/解释/洞察，而非计算

- **任务类型 1：规划步进（Planning）**
  - 输入：场景描述、分析目标、可用数据源、已有指标/结果摘要。
  - 输出：分析步骤列表、建议使用的分析框架、哪类数据需要补充。

- **任务类型 2：生成洞察（Insight generation）**
  - 输入：已完成的分析结果（指标/走势/异常等）、数据出处、场景背景。
  - 输出：可读洞察、结论、需要关注的点、下一步建议。

- **任务类型 3：用户追问（Conversation）**
  - 输入：历史问答上下文、最新分析结果、追问内容。
  - 输出：回答（可附带需要补充的指标/数据），或生成进一步的分析建议。

### 3.2 调用内容结构（示例）

- 系统角色（System）：说明“你是一个分析助手，不参与计算，只负责解释和建议”。
- 用户角色（User）：包含场景定义 + 本次分析目的 + 数据来源说明 + 本次分析结果摘要。
- 约束
  - 指定“不得用 LLM 数值推算”，只允许利用已计算结果。
  - 说明数据来源必须在回答中标注出处（“数据来源：XXX”）。

## 4. 扩展性（可新增场景/框架/数据源）

### 4.1 场景可扩展

- 场景视为“分析配置模版”，包含：描述、默认提示词、推荐框架、可用数据源列表、权限信息。
- 新增场景只需提供元信息和模板（可在 UI 里通过 JSON/表单创建）。

### 4.2 分析框架可扩展

- 框架本质为一套 “步骤规范 + 参数模板 + 计算逻辑组装器”。
- 新框架可通过插件式方式注册（本地代码/远程脚本），实现：
  - 计算组件（可执行的代码片段，例如：趋势线拟合、归因分析、漏斗计算）。
  - LLM 输出模板（用于呈现结果、洞察的文本生成提示模板）。

### 4.3 数据源可扩展

- 数据源适配器模式：每个数据源实现统一接口（fetch、describe、validate、cacheKey）。
- 支持新增：API、SQL、指标平台、数据湖、文件、监控数据等。
- 数据来源说明必须包含“来源标识 + 获取条件 + 时间戳”，方便在输出中引用。

## 5. Skill 与场景 / 分析框架的关系

- **Skill（能力模块）**：封装可复用的能力（如：指标计算、时间序列差分、异常检测、归因分析、数据预处理、图表渲染等）。
- **场景（Scene）**：用户的业务上下文（如：用户行为分析、营收分析、运维异常分析），它会组合若干 Skill 作为“分析能力集合”。
- **分析框架（Framework）**：定义分析思路和流程（如：漏斗分析框架、趋势分析框架、对比分析框架），它会调用一个或多个 Skill 来执行具体步骤。

关系简述：**场景 -> 选择/推荐分析框架 -> 框架调用 Skill**。

## 6. 三个核心问题及解决思路

### 核心问题 1：如何让系统“根据场景自动规划分析步骤”？

**思路**：

- 将每个场景预定义为“模板 + 规则集（可编写/编辑）”，里面包含可用数据、目标指标、优先级。
- 调用 LLM（仅作规划）生成“待执行步骤清单”，并将该清单映射为可执行的模块（Skill + 数据源）。
- 将 LLM 规划与系统内置规则做双重校验：如发现关键数据缺失，返回“缺少数据”给调度层并触发数据获取。

### 核心问题 2：如何保证“LLM 不参与计算”，又能生成有价值的洞察？

**思路**：

- 设计明确规范：所有涉及数值/指标的计算由系统执行，结果形成“结构化输出”（表格/JSON）。
- 传给 LLM 的输入只包含“计算后的结构化结果 + 数据出处 + 业务背景 + 目标问题”。
- LLM 仅负责基于结果生成洞察描述、对比结论、注意点、建议行动项。

### 核心问题 3：如何让系统“自动获取数据并标注出处”？

**思路**：

- 为每个数据来源建立“数据元信息（Data Provenance）”结构：包括原始接口/表名、查询条件、更新时间、相关指标说明。
- 分析流程中统一传递数据 + 元信息，输出层在呈现洞察时附带“数据来源：XXX（字段/接口/SQL）”。
- 若 LLM 发现结果缺数据，系统通过“缺失检查模块”触发补数据流程，自动调用对应适配器获取新数据，并在后续输出中标记新增来源。
